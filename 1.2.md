# Exercise 3.2 — Deployment + ReplicaSet

This exercise introduces the Kubernetes Deployment controller and the ReplicaSet it manages. You'll learn how Kubernetes maintains desired state, self-heals failed Pods, scales applications, performs rolling updates, and supports rollbacks.

---

## Create a Deployment

```bash
cat <<EOF | kubectl apply -f -
apiVersion: apps/v1
kind: Deployment
metadata:
  name: podinfo
  namespace: default
spec:
  replicas: 2

  selector:
    matchLabels:
      app: podinfo

  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0

  template:
    metadata:
      labels:
        app: podinfo
        version: "6.6"

    spec:
      containers:
      - name: podinfo
        image: ghcr.io/stefanprodan/podinfo:6.6.0

        ports:
        - containerPort: 9898
          name: http

        env:
        - name: PODINFO_UI_COLOR
          value: "#4caf50"

        resources:
          requests:
            cpu: "50m"
            memory: "64Mi"
          limits:
            cpu: "200m"
            memory: "128Mi"

        livenessProbe:
          httpGet:
            path: /healthz
            port: 9898
          initialDelaySeconds: 5

        readinessProbe:
          httpGet:
            path: /readyz
            port: 9898
          initialDelaySeconds: 3
EOF
```

## Demonstrate Self-Healing

Delete one of the Pods:

```bash
kubectl delete pod $(kubectl get pod -l app=podinfo -o name | head -1)
```

Watch Kubernetes create a replacement automatically:

```bash
kubectl get pods -w
```

Expected behavior:

```text
Pod deleted
    ↓
ReplicaSet detects missing replica
    ↓
New Pod created automatically
```

This is Kubernetes self-healing.

## Scale the Application

Increase replicas from 2 to 4:

```bash
kubectl scale deployment podinfo --replicas=4
```

Verify:

```bash
kubectl get pods
```

Expected:

```text
4 Pods Running
```

Check the Deployment:

```bash
kubectl get deployment podinfo
```

Expected:

```text
READY   UP-TO-DATE   AVAILABLE
4/4     4            4
```

## Perform a Rolling Update

Update the environment variable:

```bash
kubectl set env deployment/podinfo PODINFO_UI_COLOR="#2196f3"
```

Watch rollout progress:

```bash
kubectl rollout status deployment/podinfo
```

Observe Pods:

```bash
kubectl get pods -w
```

Expected sequence:

```text
Old Pod remains running
    ↓
New Pod starts
    ↓
New Pod becomes Ready
    ↓
Old Pod removed
```

Because:

```yaml
maxSurge: 1
maxUnavailable: 0
```

Kubernetes performs a zero-downtime rolling update.

## Roll Back the Deployment

Undo the last rollout:

```bash
kubectl rollout undo deployment/podinfo
```

Watch rollback status:

```bash
kubectl rollout status deployment/podinfo
```

Verify rollout history again:

```bash
kubectl rollout history deployment/podinfo
```

### Pod vs Deployment

| Feature          | Pod | Deployment |
| ---------------- | --- | ---------- |
| Self-healing     | ❌   | ✅          |
| Scaling          | ❌   | ✅          |
| Rolling updates  | ❌   | ✅          |
| Rollbacks        | ❌   | ✅          |
| Production-ready | ❌   | ✅          |

A standalone Pod is temporary. A Deployment continuously ensures the desired number of Pods are running and healthy.
