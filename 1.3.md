# Exercise 1.3 — Services and DNS

In this exercise, you'll learn how Kubernetes Services provide stable networking for Pods and how DNS enables service discovery inside the cluster.

> **Prerequisite:** Complete Exercise 3.2 first. The Service created here targets the `podinfo` Pods created by the Deployment.

---

## Create a ClusterIP Service

A ClusterIP Service is only accessible from inside the Kubernetes cluster.

```bash
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Service
metadata:
  name: podinfo
spec:
  selector:
    app: podinfo

  ports:
  - port: 80
    targetPort: 9898

  type: ClusterIP
EOF
```

Verify:

```bash
kubectl get svc
```

Expected:

```text
NAME      TYPE        CLUSTER-IP
podinfo   ClusterIP   10.x.x.x
```

---

## Create a NodePort Service

A NodePort Service exposes the application on a port of every Kubernetes node.

```bash
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Service
metadata:
  name: podinfo-nodeport
spec:
  selector:
    app: podinfo

  ports:
  - port: 9898
    targetPort: 9898
    nodePort: 30001

  type: NodePort
EOF
```

Verify:

```bash
kubectl get svc
```

Expected:

```text
NAME               TYPE       PORT(S)
podinfo            ClusterIP  80/TCP
podinfo-nodeport   NodePort   9898:30001/TCP
```

## Test DNS Inside the Cluster

Launch a temporary curl Pod and access the Service using its DNS name:

```bash
kubectl run curl \
  --image=curlimages/curl \
  --rm -it \
  --restart=Never -- \
  curl http://podinfo.default.svc.cluster.local
```

Expected response:

```html
<html>
...
podinfo
...
</html>
```

## Understanding Kubernetes DNS

Every Service automatically receives a DNS entry.

Format:

```text
<SERVICE>.<NAMESPACE>.svc.cluster.local
```

For this Service:

```text
podinfo.default.svc.cluster.local
```

Traffic flow:

```text
DNS Name
    ↓
Service
    ↓
Podinfo Pods
```

## Test Access from Outside the Cluster

Use the public IP of your EC2 instance:

```bash
curl http://<your-ec2-ip>:30001
```

Example:

```bash
curl http://54.123.45.67:30001
```

Expected response:

```html
<html>
...
podinfo
...
</html>
```

If access fails, verify that your EC2 Security Group allows:

```text
TCP 30001
```

## View Service Endpoints

A Service routes traffic to matching Pods through Endpoints.

```bash
kubectl get endpoints podinfo
```

Example:

```text
NAME      ENDPOINTS
podinfo   10.42.0.5:9898,10.42.1.7:9898
```

These are the actual Pod IPs receiving traffic.

## Inspect kube-proxy Rules

Kubernetes Services are implemented through networking rules managed by kube-proxy.

Inspect the generated iptables rules:

```bash
sudo iptables -t nat -L KUBE-SERVICES | grep podinfo
```

You should see entries similar to:

```text
KUBE-SVC-XXXXX
```

These rules redirect Service traffic to backend Pods.

## Network Flow

### ClusterIP

```text
Pod
 ↓
Service DNS
 ↓
ClusterIP
 ↓
Backend Pods
```

Accessible only from inside the cluster.

---

### NodePort

```text
Internet
 ↓
EC2 Public IP
 ↓
NodePort (30001)
 ↓
Service
 ↓
Pods
```

Accessible from outside the cluster.

### Service Types Overview

| Type         | Accessible From              | Common Usage       |
| ------------ | ---------------------------- | ------------------ |
| ClusterIP    | Inside cluster only          | Internal APIs      |
| NodePort     | Outside cluster via node IP  | Learning & testing |
| LoadBalancer | Cloud provider load balancer | Production         |
| ExternalName | External DNS target          | External services  |

### Kubernetes Learning Path

```text
Pod
 ↓
Deployment
 ↓
Service
 ↓
Ingress
 ↓
Load Balancer
```

Services are the foundation that Ingress controllers and Load Balancers build upon.
